# Dynamic room scaling and camera follow for play mode

The living room environment now scales to enclose tracks of any size. `buildLivingRoom()` accepts an optional `RoomExtent` parameter; the renderer computes the track's XZ bounding box on every `rebuildTrack` call, pads it by 8 units, and disposes/rebuilds the room at the new dimensions. Furniture, rug, lights, and window positions all scale proportionally. A new `followCar` method lerps the camera target toward the car each frame during play mode, and `resetCameraToTrack` restores the overview when returning to build mode.

**Watch for:** `followCar` has a coordinate-axis swap bug (confirmed) ã it feeds TrackFrame grid-space pos directly to the Three.js camera target without the Y/Z remapping that every other grid-to-Three.js conversion in the codebase performs. The camera will drift vertically instead of following the car's depth movement, increasingly wrong on tracks with large Y-extent or height variation.

## High-level view

The room-scaling design computes a bounding box of all piece entry positions, derives a `roomHalf` (clamped to a minimum of 16), and rebuilds the entire environment group from scratch each time the track changes. Wall height scales proportionally (9/16 ratio), fog range scales with room size, and the floor size is `max(72, roomHalf * 4.5)`. The rebuild also fires redundantly on camera-only resets (mode switch back to build), disposing and recreating identical geometry.

The camera follow uses exponential interpolation (`1 - exp(-4 * dt)`) for frame-rate-independent smoothing. However, the implementation passes `sample.pos` (in grid coordinate order: x, y=forward, z=height) directly to `cameraTarget` (in Three.js order: x, y=height, z=forward), producing a Y/Z axis swap that will cause the camera to track upward instead of forward on most tracks.

The furniture/light scaling uniformly multiplies all positions by `roomHalf / DEFAULT_ROOM_HALF`, keeping items proportionally placed against the walls. The furniture objects themselves are not scaled ã only their positions ã so a very large room will have normal-sized furniture near distant walls.

## followCar coordinate mapping is inverted

Every grid-to-Three.js conversion in the renderer swaps Y and Z: `placeCar` does `new THREE.Vector3(frame.pos.x, frame.pos.z, frame.pos.y)`, the mesh builder does `(b.wx - a.wx, b.wz - a.wz, b.wy - a.wy)`, and `_recenterCamera` does `cameraTarget.set(cx/n, cz/n, cy/n)`.

`followCar` breaks this convention:

```typescript
followCar(carPos: { x: number; y: number; z: number }, dt: number): void {
  const lerpFactor = 1 - Math.exp(-4 * dt);
  this.cameraTarget.x += (carPos.x - this.cameraTarget.x) * lerpFactor;
  this.cameraTarget.y += (carPos.y - this.cameraTarget.y) * lerpFactor; // grid y (forward) -> Three.js height
  this.cameraTarget.z += (carPos.z - this.cameraTarget.z) * lerpFactor; // grid z (height) -> Three.js depth
}
```

The correct mapping is `cameraTarget.y <- carPos.z` and `cameraTarget.z <- carPos.y`. On a flat track the height is ~0 so the error may be subtle (camera stays near ground regardless), but any track with Y-extent >2 or height variation will show the camera drifting vertically and failing to follow the car's lateral/forward movement. **Confidence: confirmed** ã traced through `localToWorld`, `buildFrame`, `placeCar`, and `_recenterCamera`.

## Redundant dispose/rebuild on mode switch

`resetCameraToTrack` -> `_recenterCamera` -> `_rebuildEnvironmentForTrack` disposes every mesh in the room group and builds a new one with identical dimensions. This happens every time the player exits play mode. The disposal traverses the entire group calling `geometry.dispose()` and `material.dispose()` on dozens of objects, then `buildLivingRoom` re-creates them all.

A simple guard (`if (roomHalf === this._currentRoomHalf) return`) or splitting `_recenterCamera` so the camera-only path doesn't trigger a rebuild would fix this. **Confidence: confirmed** ã the call chain is `resetCameraToTrack` -> `_recenterCamera` -> `_rebuildEnvironmentForTrack` with no size-change check.

## Shadow frustum doesn't scale with room

The directional light's shadow camera is configured once in `_addLights`:

```
sun.shadow.camera.left = -18;
sun.shadow.camera.right = 18;
sun.shadow.camera.top = 18;
```

When `roomHalf` exceeds 18 (any track extending more than 10 units from center), pieces beyond the shadow frustum will have no shadows. The room walls and furniture scaled to the edges will also lack shadows. The shadow map would need to scale with the room, or use a CSM/follow approach. **Confidence: confirmed** ã `_addLights` runs once in the constructor and is never revisited by `_rebuildEnvironmentForTrack`.
